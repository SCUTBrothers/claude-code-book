# 《深入浅出 Claude Code》执行方案

## 1. 文档定位

本文件是《深入浅出 Claude Code》的主控执行方案。它不直接承担章节正文写作，而是定义整本书的生产流程、任务编排、提示词约束、审查门禁、资料沉淀方式，以及 LaTeX 交付标准。

## 2. 总目标与边界

### 2.1 总目标

- 以 `./claude-code-source` 为主语料，系统解析 Claude Code 源码。
- 通过 Chrome 中的 Claude 对话循环任务机制，持续产出结构化章节内容。
- 将所有通过审查的章节整理为 LaTeX，并最终编译为 PDF。
- 形成一套可复用的“源码成书”流水线，而不是一次性写作行为。

### 2.2 输出物

- 书籍主控方案：`./book.md`
- 过程文档：`./docs`
- LaTeX 书稿：`./book`
- 最终 PDF：`./book/build`

### 2.3 非目标

- 不追求覆盖每一个源文件；重点覆盖核心设计与关键实现路径。
- 不把外部博客当作一手证据；源码永远优先于二手解读。
- 不让 Claude 在单轮上下文中“吃掉整个仓库”；必须进行模块化切分。
- 仓库访问与发布策略不写入提示词正文，避免把真实 remote 或个人协作指令暴露到可追溯产物中。

### 2.4 硬约束

- 所有技术结论必须明确区分：源码事实、合理推断、外部资料。
- 任一章节在进入下一阶段前，必须完成计划、写作、审查、LaTeX、编译检查五个动作。
- 不允许跳过 review。
- 浏览器会话不是事实来源，所有中间状态必须落盘到本地文件。
- 当静态阅读无法可靠解释某个机制时，允许通过运行代码、注入日志、观测行为来补充验证，但必须记录注入点、运行条件、观测结果和回滚方式。
- 动态验证只能作为源码分析的补强手段，最终结论仍必须回连到源码结构与实现位置。

## 3. 当前项目基线

结合当前仓库可确认如下事实：

- 目标源码位于 `claude-code-source/`，版本基线为 `2.1.88`。
- 仓库已包含源码恢复说明、构建说明、模块分布说明。
- 当前存在 `book/` 与 `docs/` 目录，可作为书稿与过程资产目录。

这意味着本项目适合采用“源码索引 -> 模块综述 -> 章节 dossier -> 分节写作 -> LaTeX 编排”的分层生产法，而不适合一次性长提示直接出全书。

## 4. 《深入浅出 Claude Code》建议结构（v2）

### 4.1 行文风格借鉴原则

本书的行文可以借鉴优秀计算机书籍中“由浅入深、先立问题、再讲机制、最后落到实现”的写法，但不能照搬任何一本书的目录或语气。建议吸收以下共性：

- 先回答读者为什么要关心这个问题，再进入实现细节。
- 先建立心智模型，再下钻到源码文件、类型、调用链。
- 每章只解决一类核心问题，不把多个主题揉成杂糅综述。
- 图示、流程图、对照表服务于理解，不作为装饰。
- 章节末尾给出稳定结论、边界与过渡，而不是口号式总结。

不建议采用的写法：

- 为了“浅出”而过度儿童化、过度故事化。
- 为了“高级感”而强行写成论文腔。
- 为了“像书”而加入大量无信息的铺垫段落。

### 4.2 前置书稿组件

这本书不应从正文章节直接开始，前置部分应单独设计：

| 组件     | 作用                             | 说明                                 |
| -------- | -------------------------------- | ------------------------------------ |
| 扉页     | 展示书名、副标题、版本信息       | 书名统一为《深入浅出 Claude Code》   |
| 推荐语页 | 展示若干角色化评语样稿           | 仅作气质与版式参考，不伪装为真实背书 |

| 导读     | 告诉读者如何阅读本书             | 交代章节关系、术语体系、阅读路径     |

### 4.4 推荐语设计与合规边界

可以加入“推荐语页”，但必须明确区分“真实背书”和“角色化样稿”：

- 未获得授权前，不使用真实工程师姓名、真实公司职位、真实机构名义。
- 在方案和书稿中统一标注：`以下推荐语为角色化样稿，不代表真实人物或机构背书。`
- 如果后续获得真实审阅意见，再用正式授权文本替换样稿。

建议采用的角色标签：

- 某大厂 AI 基础设施工程师
- 资深终端工具工程师
- LLM Agent 平台负责人
- 编程工具产品工程师

可先拟写 3 到 5 条样稿，控制长度在 50 到 120 字之间，语气应克制、专业、具体。

样稿示例：

- “这本书真正难得的地方，不在于介绍 Claude Code 能做什么，而在于把它为什么能这样工作讲清楚了。它把会话、工具、权限、上下文和运行时连接成了一个完整系统。”（角色化样稿，某大厂 AI 基础设施工程师）
- “很多人会用 Agent 工具，但很少有人从工程实现层面理解它的边界和代价。这本书适合已经在一线写代码、正在思考工具系统化问题的工程师。”（角色化样稿，资深终端工具工程师）
- “如果你关心的不只是产品体验，而是一个 Agent 编程系统如何被构建、约束、扩展和治理，这本书提供了一条相对扎实的源码阅读路径。”（角色化样稿，LLM Agent 平台负责人）

## 7. Chrome 循环任务机制

## 7.1 设计原则

- Chrome 只负责驱动 Claude Web 的持续交互，不负责保存真实状态。
- 任务状态、提示词、章节计划、审查结果必须以文件为准。
- 浏览器上下文必须定期清空或重建，避免“会话记忆漂移”。

## 7.2 任务状态机

每个任务必须在本地队列中有状态：

```text
queued
-> dispatched
-> answered
-> self_checked
-> reviewed
-> revised
-> texed
-> compiled
-> page_checked
-> done
```

若失败，则进入：

```text
blocked
-> retry_pending
-> dispatched
```

## 7.3 单个任务包格式

建议队列项至少包含：

```json
{
  "task_id": "ch05-sec02-draft-v1",
  "chapter": "05",
  "section": "02",
  "stage": "draft",
  "objective": "解释 compact 机制的触发条件、压缩策略与副作用",
  "scope_files": [
    "src/services/compact/compact.ts",
    "src/services/compact/autoCompact.ts",
    "src/services/compact/prompt.ts"
  ],
  "required_outputs": ["知识点大纲", "小节初稿", "引用点清单", "待核实问题"],
  "acceptance_checks": [
    "必须区分源码事实与推断",
    "必须给出文件路径与符号名",
    "必须标记未证实结论"
  ]
}
```

## 7.4 循环流程

1. 调度器从 `docs/queue/tasks.jsonl` 取一个 `queued` 任务。
2. 组装稳定系统提示词、章节上下文摘要、任务包。
3. 通过 Chrome 将任务发送给 Claude。
4. Claude 输出严格结构化结果。
5. 本地校验器检查输出完整性、格式、引用、术语一致性。
6. 通过则落盘；失败则生成修复任务并重试。
7. 每完成 3 到 5 个任务，关闭当前浏览器会话，重新创建新会话，只注入压缩后的上下文摘要。

## 7.5 防漂移规则

- 单次任务只允许覆盖一个小节或一个明确子目标。
- 单次任务的源码输入应限制在一个模块簇，不要跨越过多目录。
- 如需跨章引用，必须先读取 `glossary.md` 与 `concept-map.md`。
- Claude 不得“顺手改大纲”；所有结构调整必须走单独的 outline 任务。

## 7.6 静态分析与动态验证

本书的源码研究默认采用“静态阅读优先，动态验证补强”的方法。

优先顺序如下：

1. 先读文件、类型、调用链、注释与构建脚本。
2. 若仍无法可靠解释机制，再设计最小化运行实验。
3. 通过日志、埋点、输入输出对照、状态变化来观察真实行为。
4. 再把观察结果回映到源码，确认是哪个模块、分支、状态转换在起作用。

适合发起动态验证的场景：

- 单靠静态阅读难以判断真实执行顺序。
- 代码跨越多个模块，调用链太长，难以一次性建立稳定心智模型。
- 存在运行时状态、缓存、异步调度、权限门禁或事件驱动逻辑。
- 静态分析与实际表现之间出现冲突，需要用实验纠偏。

不适合发起动态验证的场景：

- 仅为了“多做一点工作”而盲目跑全项目。
- 还没建立最基本的静态假设，就直接依赖日志堆数据。
- 改动过大、埋点过多，导致行为被 instrumentation 本身污染。

## 7.7 动态验证标准流程

当某个机制需要动态验证时，建议使用以下流程：

1. 在 `*-probe.md` 中写清楚待验证问题、初始假设、目标入口、预期现象。
2. 选择最小可复现实验，而不是直接启动整套复杂流程。
3. 在关键边界位置注入临时日志或轻量埋点，优先记录输入、状态、分支、输出。
4. 运行目标流程，保存日志、终端输出、关键中间结果。
5. 将日志整理为 `*-trace.md`，按时间顺序解释发生了什么。
6. 把观察结果映射回源码中的函数、模块、状态机或权限判断。
7. 删除或隔离临时埋点，避免污染后续分析与正式书稿。

动态验证的记录必须至少包含：

- 验证问题
- 运行入口
- 关键文件与埋点位置
- 输入条件
- 观测到的日志或行为
- 与静态分析相比新增了什么结论
- 仍未解释清楚的部分

说明：

- 日志不是结论本身，日志只是帮助定位结论。
- 如果日志只能说明“发生了”，但不能说明“为什么发生”，仍然要回到源码。
- 书里可以引用动态验证得到的认识，但表达时必须说明它来自运行观测，而非纯静态阅读。

## 8. 多 Agent 协作方式

“多个 team agent 共同运行”不能理解为多个 agent 同时向同一正文文件乱写。正确做法是分工并隔离写入责任：

| Agent 角色           | 职责                                   | 输出                       |
| -------------------- | -------------------------------------- | -------------------------- |
| Outline Agent        | 设计章节结构与叙事主线                 | `outline.md`, `backlog.md` |
| Source Analyst Agent | 读取源码、提取实现路径、列证据锚点     | `*-sources.md`             |
| Probe Agent          | 设计并执行动态验证，整理埋点与日志结论 | `*-probe.md`, `*-trace.md` |
| Research Agent       | 查询外部资料、提炼背景知识与对照材料   | `external-notes.md`        |
| Writer Agent         | 基于已审过的输入写初稿                 | `*-draft.md`               |
| Review Agent         | 审查逻辑、证据、表达与一致性           | `*-review.md`              |
| LaTeX Agent          | 把通过审核的内容转成 `.tex` 并修排版   | `book/chapters/*.tex`      |

执行规则：

- 多 agent 可以并行准备素材，但同一时刻只能有一个 agent 负责某个目标正文文件的最终写入。
- Review Agent 与 Writer Agent 必须分离，避免自我放行。
- LaTeX Agent 只接收“审查通过”的稿件，不直接处理未定稿内容。

## 8.1 Sub-agent 使用原则

源码分析阶段应优先把跨模块、易污染上下文的问题交给 `sub-agent` 处理，再由主 agent 汇总。这样做的目的不是炫技，而是降低主会话被大量局部细节淹没的概率。

适合交给 `sub-agent` 的任务：

- 某个跨模块调用链的追踪与整理
- 某个服务簇的职责边界梳理
- 某个运行实验的日志整理与结论提炼
- 某个章节 dossier 的证据收集

不适合交给 `sub-agent` 的任务：

- 最终章节定稿
- 跨多个章节的大纲改写
- 尚未界定边界的开放式大问题

推荐用法：

1. 主 agent 先定义问题边界与交付格式。
2. 把一个明确子问题交给单独 `sub-agent`。
3. `sub-agent` 只返回结构化结果，不直接改正文。
4. 主 agent 汇总多个 `sub-agent` 的结果，统一口径后再进入写作。

这样可以显著减少分析期的上下文污染，也能避免同一轮里把静态阅读、日志追踪、写作、审查混成一锅。

## 9. Prompt Engineering 约束

## 9.1 总体约束

每轮提示词都必须遵守以下规则：

1. 先列知识点大纲，再写正文。
2. 先给写作计划，再输出内容。
3. 先列资料需求，再开始论述。
4. 先给引用候选点，再生成定稿。
5. 不能把“未读源码”当作已知事实。
6. 不能编造文件、函数、模块关系。
7. 不能把推测写成定论。
8. 不能直接复刻长代码块，优先用短片段、伪代码或结构化解释。
9. 必须显式标注不确定项：`待核实`。
10. 必须优先服从本地源码与已存档资料，而不是浏览器上下文记忆。

## 9.2 证据标签

正文草稿与审查稿中，建议给关键论断标记证据等级：

- `[S]`：源码直接可证实。
- `[I]`：基于源码结构的合理推断。
- `[E]`：外部资料补充。
- `[Q]`：待核实问题。

这一步会显著降低 Claude 把推断说成事实的概率。

## 9.3 输出格式模板

每次任务要求 Claude 使用固定结构输出：

```text
一、知识点大纲
二、本轮写作计划
三、资料收集清单
四、候选引用点
五、正文草稿
六、待核实问题
七、自审结果
```

若任一段缺失，则该轮任务视为未完成。

## 9.4 推荐系统提示词骨架

```text
你正在参与一本基于 Claude Code 源码的技术书写作。

你的首要目标不是“写得像书”，而是“写得可验证、可合并、可复审”。

请严格遵守：
1. 只依据本轮提供的源码范围、既有术语表、已存档资料作答。
2. 若证据不足，明确写“待核实”，不得补全不存在的实现细节。
3. 所有关键论断都要标注 [S]/[I]/[E]/[Q]。
4. 先输出大纲、计划、资料需求、引用点，再输出正文。
5. 正文以工程设计解释为主，不写营销语言，不写空泛赞美。
6. 如果知识结构是网状的，应主动建议图表或流程图，而不是只写线性文字。
7. 输出必须与本轮任务目标严格对齐，不得擅自扩写到其他章节。
```

## 9.5 内容生成机制

本项目不采用“单轮生成整节正文”的方式，而采用六段式生成机制。目标很明确：先保证结构，再保证事实，再保证文风。

| 阶段   | 目标     | 输入                             | 输出                         |
| ------ | -------- | -------------------------------- | ---------------------------- |
| Pass 1 | 搭骨架   | 章节目标、读者画像、源码范围     | 小节骨架与论证顺序           |
| Pass 2 | 补证据   | 源码文件、符号名、外部资料摘要   | 证据锚点与引用点             |
| Pass 3 | 动态验证 | 待核实问题 + 运行入口 + 埋点计划 | `probe/trace` 结果或跳过说明 |
| Pass 4 | 写初稿   | 骨架 + 证据包 + 验证结果         | 粗稿                         |
| Pass 5 | 重写结构 | 粗稿 + 审查意见                  | 连贯版正文                   |
| Pass 6 | 去 AI 味 | 连贯版正文 + 风格规范            | 科技文风格稿                 |
| Pass 7 | 统一术语 | 全章上下文 + 术语表 + 概念图     | 可合并定稿                   |

执行要求：

- 任何一轮都不能跨越多个阶段直接产出“终稿”。
- Pass 3 只在确有必要时启动，避免把动态验证变成默认动作。
- Pass 4 的任务是把内容写出来，不负责风格净化。
- Pass 5 的任务是消除散点堆砌、重复定义、段落跳跃。
- Pass 6 的任务是消除 AI 腔、空泛连接句、模板化总结。
- Pass 7 的任务是统一与前文术语、章节口径、图表编号。

## 9.6 章节级生成规则

每个章节、每个小节都必须先形成叙事主线，禁止把素材按“想到什么写什么”的顺序堆起来。

推荐使用以下章节展开顺序：

1. 先说明问题背景：这个模块解决什么问题。
2. 再说明系统位置：它在整个 Claude Code 中处于哪里。
3. 再说明关键机制：核心对象、流程、状态或约束是什么。
4. 再说明实现证据：落到文件、类型、函数、调用链。
5. 再说明工程权衡：为什么这样设计，代价是什么。
6. 最后说明边界与过渡：本节结论如何连接下一节。

禁止出现以下写法：

- 第一段就在列文件路径，但没有先交代问题。
- 前两段讲概念，后面突然插入实现细节，没有过渡。
- 同一小节里同时展开两个主问题，导致读者主线丢失。
- 每段都像一个独立问答，拼不成一条稳定叙事链。

## 9.7 段落级生成单元

为避免内容零散，每个自然段原则上只承载一个核心判断，并按以下顺序组织：

1. 主题句：先给出本段要说明的技术判断。
2. 机制句：解释这个判断成立的内部机制。
3. 证据句：落到源码文件、符号、调用点或资料来源。
4. 约束句：说明例外、限制、成本或边界。
5. 过渡句：必要时连接下一段，不强行总结。

说明：

- 不是每段都必须写满五句，但顺序应尽量稳定。
- 如果一段无法概括为一个中心判断，说明该段仍然过散，应继续拆分。
- 图表、表格、代码片段要服务于本段判断，不能为了“显得丰富”而硬塞。

## 9.8 科技文风格控制

本书的文风目标不是口语化教程，也不是论文腔，而是“工程师写给工程师的科技文”。

应遵守以下原则：

- 先定义术语，再持续复用同一叫法，不频繁换同义词。
- 多用判断句和解释句，少用感叹句、反问句、口号句。
- 多写因果、约束、取舍，少写空泛评价。
- 尽量把抽象概念绑定到对象、流程、状态、边界。
- 段落之间要有逻辑接缝，而不是机械加“此外”“另外”“值得一提的是”。
- 序言、导读和推荐语页的语气可以稍微更有人味，但正文必须回到克制、准确、可验证的科技文风格。

明确禁止的 AI 味表达：

- “可以说”“不难发现”“值得注意的是”反复出现。
- “首先、其次、最后”在非枚举场景下机械重复。
- 没有信息增量的总结句，如“这体现了优秀的工程思想”。
- 形容词堆叠，如“强大、灵活、优雅、完善”。
- 在未给证据时写“显然”“本质上”“可以理解为”。

## 9.9 成稿净化规则

在进入 review 之前，必须对草稿做一次“文风净化”：

1. 删除空泛过渡句。
2. 合并重复定义。
3. 把“概念复述”改成“机制解释”。
4. 把“功能罗列”改成“主线驱动”。
5. 删除不承载信息的赞美性句子。
6. 检查每段是否都能回答“这一段到底新增了什么信息”。

如果一个小节可以删掉 20% 文字而不损失信息密度，通常说明初稿仍有明显 AI 膨胀，应继续压缩。

## 10. 资料探索与联网策略

## 10.1 资料优先级

按如下顺序取材：

1. 本地源码
2. 本地 README、构建脚本、注释、类型定义
3. Anthropic 官方文档与官方博客
4. MCP 协议、Ink、Bun、Commander 等相关官方文档
5. 高质量工程博客或会议分享
6. 社区讨论

原则：

- 外部资料只能增强解释能力，不能覆盖源码事实。
- 一切“历史背景”“设计动机”“业内对照”类内容，都必须记录来源与访问日期。

## 10.2 外部资料归档规范

每条外部资料至少记录：

- 标题
- 作者
- 发布日期
- 访问日期
- URL
- 可信级别
- 可用于解释的知识点
- 不可直接采信的部分

## 10.3 建议重点调研主题

- Claude / Agent 产品形态演化
- Context Engineering 与上下文压缩
- 工具调用与权限门禁模式
- 终端 UI 架构与 Ink 的设计权衡
- MCP 生态、扩展边界与安全问题
- 多代理协作与任务分解

## 11. 每个章节/小节的标准作业流程

任一小节必须严格走完以下步骤：

1. 创建 `*-plan.md`
2. 写出知识点大纲
3. 写出本轮编写计划
4. 定义本节主线与章节位置
5. 列出需要收集的源码与外部资料
6. 标注候选引用点
7. 判断是否需要发起动态验证
8. 若需要，则创建 `*-probe.md` 并执行运行实验
9. 整理 `*-trace.md`
10. 先生成骨架稿
11. 再生成证据包
12. 产出 `*-draft.md`
13. 执行结构重写与去 AI 味净化
14. 进行 Claude 自审
15. 进行 Codex review
16. 修复问题并复审
17. 转写为 `.tex`
18. 编译检查
19. 页面级版式检查
20. 更新 `glossary.md`、`concept-map.md`、`backlog.md`

只有第 15 步真正通过，才能进入第 17 步。

## 12. 审查机制

## 12.1 强制门禁

每一轮必须经过以下门禁：

| Gate   | 目标         | 通过条件                                           |
| ------ | ------------ | -------------------------------------------------- |
| Gate A | 计划审查     | 目标、边界、资料单、引用点完整                     |
| Gate B | 事实审查     | 关键论断可追溯到源码或已登记外部资料               |
| Gate C | 验证审查     | 若使用动态验证，则埋点、日志、结论映射完整且可回溯 |
| Gate D | 结构审查     | 主线清晰，小节不是素材拼盘，段落承接自然           |
| Gate E | 表达审查     | 无明显 AI 套话，无空泛段落，无机械总结句           |
| Gate F | 一致性审查   | 与已写章节术语、定义、结论不冲突                   |
| Gate G | LaTeX 审查   | 代码块、图表、表格、引用可正常排版                 |
| Gate H | PDF 页面审查 | 无溢出、裁切、错页、图过大、表格错框               |

## 12.2 Review 执行顺序

必须采用以下顺序：

1. Claude 自审
2. 使用 `/codex:review` 审查计划、草稿或 LaTeX 改动
3. 修复所有 findings
4. 再次 review
5. findings 为 0，或仅剩明确接受的残余风险后，才允许进入下一步

如果当前环境不支持 `/codex:review` 命令，则必须执行等价的独立审查流程，标准不降低。

## 12.3 Review 检查清单

- 是否适度使用费曼技巧，而不是把所有内容都儿童化
- 是否把适合网状表达的知识点转换成图表、流程图或对照表
- 是否存在逻辑漏洞、偷换概念、前后表述不一致
- 如果用了动态验证，是否把日志观察正确映射回了源码机制
- 如果用了 `sub-agent`，主 agent 是否完成了结果归并，而不是直接拼接输出
- 是否把“流程系统”“权限系统”“多模块联动”错误地写成线性叙事
- 是否存在素材堆砌、段落主题不明、过渡生硬的问题
- 是否存在明显 AI 腔、空泛赞美、模板化总结、无信息增量句子
- 是否符合科技文写作标准：概念清楚、边界明确、论证可追溯

## 13. LaTeX 生产规范

## 13.1 技术选型

建议默认采用：

- 文档类：`ctexbook`
- 编译方式：`latexmk -xelatex`
- 代码块：`listings` 或 `tcolorbox`
- 图表：`tikz`, `pgfplots`, `forest`, `tabularx`, `booktabs`, `longtable`

默认不优先使用 `minted`，除非编译环境已稳定支持 `-shell-escape`。原因很简单：书稿生产要稳定优先，不要让语法高亮依赖额外脆弱环节。

## 13.2 版式要求

- 大标题居中，小标题左对齐并加粗。
- 代码块必须有语言标识、换行策略与适当字号。
- 长路径、长命令、长类型名必须能自动换行。
- 图表宽度不得超出正文边界。
- 表格优先采用窄列与自动换行列，避免横向爆版。

## 13.3 编译与修复

每合并一个章节后都要执行一次全书编译，而不是只编译局部章节。

推荐命令：

```bash
latexmk -xelatex -interaction=nonstopmode -file-line-error book/main.tex
```

若失败，创建独立的 `latex-fix` 任务：

- 输入：报错日志、相关 `.tex`、相关图表
- 输出：修复补丁、原因分析、回归检查结果

## 13.4 页面级检查

逐页检查以下问题：

- 文字是否越界
- 代码块是否被截断
- 图形是否过大或分辨率不足
- 表格框线、边距、换页是否异常
- 标题层级、目录、页眉页脚是否正常

## 14. 章节写作质量标准

每节正文都应至少满足以下标准：

- 读者知道“这个模块解决什么问题”
- 读者知道“它在系统中的位置”
- 读者知道“关键执行流如何串起来”
- 读者知道“实现上的权衡与限制”
- 读者知道“与相邻模块如何协作”
- 读者能通过图表、代码片段、流程描述建立稳定心智模型
- 读者读完后能复述本节主线，而不是只记住零散事实

不达标的常见表现：

- 只罗列文件名，不解释设计
- 只讲概念，不落到源码实现
- 只讲调用链，不讲约束与权衡
- 只讲局部正确，不讲系统边界
- 段落像聊天记录整理稿，而不是成体系的小节
- 每段都正确，但连起来没有章节推进感

## 15. 风险与缓解

| 风险               | 表现                                         | 缓解                                           |
| ------------------ | -------------------------------------------- | ---------------------------------------------- |
| 上下文漂移         | Claude 后续输出与前文定义不一致              | 每轮强制读取 `glossary.md` 与 `concept-map.md` |
| 幻觉式补全         | 编造函数职责、历史背景或设计意图             | 使用证据标签与 `待核实` 机制                   |
| 纯静态误判         | 调用链太长或运行时状态复杂，导致源码理解偏差 | 在必要场景下引入最小化动态验证                 |
| Chrome 自动化脆弱  | 页面结构变动导致循环中断                     | 浏览器层只做最小交互，状态机放本地             |
| 多 agent 冲突      | 多人同时改同一章节导致内容冲突               | 单文件单 owner，review 后再合并                |
| sub-agent 污染合并 | 子问题结果未经归并直接进入正文               | 主 agent 统一整合，只接收结构化输出            |
| LaTeX 反复报错     | 代码块、图表、表格导致编译不稳               | 统一模板与组件，问题单独拆为 `latex-fix`       |
| 章节重复           | 不同章节反复解释相同概念                     | 维护术语表、概念索引、首次出现位置             |

- 页面级检查通过

## 18. 最终原则

这本书的核心不是“让 Claude 一口气写完”，而是“把 Claude 变成一个受约束的写作工位”。真正决定质量的，不是模型一次回答有多长，而是：

- 任务切分是否正确
- 提示词是否可验证
- 审查是否独立且严格
- 资料是否分级管理
- LaTeX 交付是否工程化

执行时应始终坚持一句话：先把流程做对，再把篇幅做大。
